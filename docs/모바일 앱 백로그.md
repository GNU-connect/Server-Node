# 모바일 앱 백로그

## 기준

- 원본 시나리오: [유저 시나리오.md](./유저%20시나리오.md)
- 목표: 카카오톡 챗봇에서 제공하는 기능을 모바일 앱에서도 모두 사용할 수 있게 한다.
- 현재 앱 구현 범위: 학식, 셔틀버스
- 추가 구현 필요 범위: 사용자 설정/내 정보, 학교 공지사항, 학과 공지사항, 학사일정

## 우선순위

- P0: 챗봇 기능을 앱에서 사용하기 위한 필수 기반 또는 핵심 조회 기능
- P1: 사용성을 높이는 필수 화면 상태, 개인화, 탐색 기능
- P2: 운영 편의, 품질 개선, 확장 기능

## Epic 1. 모바일 앱 정보 구조와 공통 기반

### Story 1-1. 사용자는 주요 학교 생활 정보를 하단 탭에서 이동할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 하단 탭에서 `학식`, `셔틀`, `공지`, `학사일정`, `내 정보` 화면으로 이동할 수 있다.
- 기존 학식/셔틀 화면은 유지된다.
- 각 탭은 앱 전체 디자인 시스템의 색상, 타이포그래피, 간격 규칙을 따른다.

**Tasks**

- `mobile/app/(tabs)/_layout.tsx`에 공지, 학사일정, 내 정보 탭 추가
- 각 신규 탭의 기본 라우트 파일 생성
- 탭 아이콘과 탭 라벨 정의
- 화면별 공통 로딩/오류/빈 상태 컴포넌트 후보 정리

### Story 1-2. 사용자는 앱에서 API 오류와 빈 상태를 일관되게 이해할 수 있다

**우선순위:** P1

**Acceptance Criteria**

- API 호출 실패 시 화면별 오류 문구를 보여준다.
- 데이터가 없는 경우 오류와 구분되는 빈 상태를 보여준다.
- 사용자는 실패한 조회를 다시 시도할 수 있다.

**Tasks**

- 공통 `EmptyState`, `ErrorState`, `LoadingState` 컴포넌트 추가
- 학식/셔틀 화면의 기존 상태 UI를 공통 컴포넌트로 점진 정리
- 재시도 버튼 또는 새로고침 액션 패턴 정의

### Story 1-3. 앱은 주요 조회 조건을 저장하고 다음 실행 시 복원할 수 있다

**우선순위:** P1

**Acceptance Criteria**

- 학식 화면에서 사용자가 선택한 캠퍼스와 식당이 앱에 저장된다.
- 앱을 다시 열거나 학식 화면에 재진입하면 저장된 캠퍼스와 식당이 기본 선택된다.
- 셔틀버스 화면에서 사용자가 선택한 노선이 앱에 저장된다.
- 앱을 다시 열거나 셔틀버스 화면에 재진입하면 저장된 노선이 기본 선택된다.
- 저장된 캠퍼스, 식당, 노선이 더 이상 유효하지 않으면 화면의 기본 선택값으로 fallback 된다.
- 저장 상태를 불러오지 못해도 학식/셔틀 조회 기능은 사용할 수 있다.

**Tasks**

- 조회 조건 저장소 설계: 학식 캠퍼스 ID, 식당 ID, 셔틀 노선 ID
- `mobile` 로컬 저장소 유틸 또는 훅 추가
- 학식 화면의 캠퍼스/식당 선택 변경 시 저장 처리
- 학식 화면 진입 시 저장된 캠퍼스/식당 복원 및 유효성 검증
- 셔틀 화면의 노선 선택 변경 시 저장 처리
- 셔틀 화면 진입 시 저장된 노선 복원 및 유효성 검증
- 저장값이 없거나 유효하지 않은 경우 기존 기본 선택 로직 유지
- 조회 조건 저장/복원 테스트 추가

## Epic 2. 사용자 설정과 내 정보

### Story 2-1. 사용자는 앱에서 저장된 내 프로필을 조회할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 내 정보 화면에서 저장된 캠퍼스와 학과를 확인할 수 있다.
- 저장된 정보가 없으면 캠퍼스/학과가 미설정 상태로 표시된다.
- 프로필을 불러오지 못하면 오류 상태를 보여준다.

**Tasks**

- 앱용 프로필 조회 API 설계: `GET /users/profile`
- 백엔드 native controller 및 response DTO 추가
- `mobile/services/userApi.ts` 추가
- `mobile/app/(tabs)/profile.tsx` 화면 추가
- 미설정 상태와 설정 변경 진입 버튼 구현
- 프로필 조회 테스트 추가

### Story 2-2. 사용자는 앱에서 캠퍼스와 학과를 등록하거나 변경할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 사용자는 캠퍼스를 선택할 수 있다.
- 사용자는 선택한 캠퍼스 기준으로 단과대학을 선택할 수 있다.
- 사용자는 선택한 단과대학 기준으로 학과를 선택할 수 있다.
- 저장 성공 시 내 정보 화면에서 변경된 캠퍼스와 학과가 보인다.
- 캠퍼스/단과대학/학과 정보를 찾을 수 없으면 오류 상태를 보여준다.

**Tasks**

- 앱용 단과대학 목록 API 설계: `GET /colleges?campusId=&page=`
- 앱용 학과 목록 API 설계: `GET /departments?campusId=&collegeId=&page=`
- 앱용 학과 저장 API 설계: `PUT /users/department`
- 백엔드 native controller 및 DTO 추가
- `mobile/services/userApi.ts`에 캠퍼스/단과대학/학과/저장 함수 추가
- 캠퍼스 선택 화면 또는 바텀시트 구현
- 단과대학 선택 화면 구현
- 학과 선택 화면 구현
- 저장 완료 후 프로필 캐시/상태 갱신
- 페이지네이션 또는 무한 스크롤 처리
- 등록/변경 플로우 테스트 추가

### Story 2-3. 사용자는 첫 사용 시 개인화 설정을 안내받을 수 있다

**우선순위:** P1

**Acceptance Criteria**

- 프로필에 캠퍼스 또는 학과가 없으면 내 정보 화면에서 설정을 유도한다.
- 학과 공지사항처럼 학과 정보가 필요한 기능에서 설정 화면으로 이동할 수 있다.
- 설정하지 않아도 학식/셔틀/학교 공지사항/학사일정은 사용할 수 있다.

**Tasks**

- 프로필 미설정 판별 로직 추가
- 학과 공지 화면에서 학과 설정 CTA 추가
- 설정 완료 후 원래 화면으로 돌아오는 흐름 구현

## Epic 3. 공지사항

### Story 3-1. 사용자는 앱에서 학교 공지사항을 카테고리별로 조회할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 앱은 `기관`, `채용`, `장학`, `외부기관 행사`, `학사` 카테고리의 학교 공지사항을 보여준다.
- 각 카테고리는 최신 공지사항을 최대 4개까지 보여준다.
- 전체 표시 건수는 최대 20개이다.
- 공지사항이 없으면 “현재 등록된 공지사항이 없어요” 상태를 보여준다.
- 사용자는 공지사항을 눌러 원문을 열 수 있다.

**Tasks**

- 앱용 학교 공지사항 API 설계: `GET /notices/university`
- 공지 response DTO에 카테고리, 제목, 작성일, 원문 URL 또는 원문 식별자 포함
- 백엔드 native controller 추가
- `mobile/services/noticeApi.ts` 추가
- 공지 탭 기본 화면 구현
- 학교/학과 공지 전환 탭 또는 segmented control 구현
- 카테고리별 공지 리스트 UI 구현
- 원문 열기 동작 구현
- 공지사항 없음/호출 실패 상태 구현
- 학교 공지 조회 테스트 추가

### Story 3-2. 사용자는 앱에서 내 학과 공지사항을 조회할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 학과 정보가 등록되어 있으면 내 학과 최신 공지사항을 볼 수 있다.
- 관련 상위 학과 공지사항이 있으면 함께 보여준다.
- 공지 분류별 최신 글을 최대 4개씩 보여준다.
- 학과 정보가 없으면 학과 등록 안내를 보여준다.
- 공지사항이 없으면 “현재 등록된 공지사항이 없어요” 상태를 보여준다.

**Tasks**

- 앱용 학과 공지사항 API 설계: `GET /notices/department`
- 인증 사용자 기준 학과 조회 처리
- 공지 response DTO에 학과/상위 학과 출처 정보 포함 여부 결정
- `noticeApi.ts`에 학과 공지 조회 함수 추가
- 학과 공지 화면 구현
- 학과 미설정 안내와 설정 화면 이동 구현
- 학과 공지 조회 테스트 추가

### Story 3-3. 사용자는 공지사항을 탐색하기 쉽도록 날짜와 출처를 확인할 수 있다

**우선순위:** P1

**Acceptance Criteria**

- 공지 카드 또는 리스트 항목에 제목, 카테고리, 작성일을 보여준다.
- 학과 공지는 가능하면 학과명 또는 상위 학과명을 함께 보여준다.
- 긴 제목도 화면을 깨지 않고 표시된다.

**Tasks**

- 공지 리스트 아이템 컴포넌트 구현
- 작성일 포맷 유틸 추가
- 긴 제목 말줄임/줄바꿈 정책 적용
- 접근성 라벨 적용

## Epic 4. 학식

### Story 4-1. 사용자는 앱에서 캠퍼스별 식당과 식단을 조회할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 앱은 캠퍼스 목록을 불러오고 기본 캠퍼스를 선택한다.
- 캠퍼스를 바꾸면 해당 캠퍼스의 식당 목록이 바뀐다.
- 사용자는 오늘부터 7일간 날짜를 선택할 수 있다.
- 사용자는 아침, 점심, 저녁 중 하나를 선택할 수 있다.
- 메뉴는 분류별로 묶어서 표시된다.

**Tasks**

- 기존 학식 화면의 시나리오 충족 여부 점검
- 식당이 없는 캠퍼스의 빈 상태 보강
- 캠퍼스/식당/메뉴 API 실패 상태 재시도 추가
- 식단 조회 단위 테스트 또는 화면 테스트 추가

### Story 4-2. 앱은 처음 열렸을 때 현재 시간에 맞는 끼니를 자동 선택한다

**우선순위:** P0

**Acceptance Criteria**

- 19:00부터 다음 날 09:29까지는 `아침`이 선택된다.
- 09:30부터 13:29까지는 `점심`이 선택된다.
- 13:30부터 18:59까지는 `저녁`이 선택된다.
- 앱의 자동 선택은 휴대폰 현재 시간 기준으로 동작한다.

**Tasks**

- `mobile/utils/dietTime.ts` 기준 시나리오와 일치 여부 점검
- 경계 시간 테스트 추가
- 학식 화면 초기 선택 동작 테스트 추가

### Story 4-3. 사용자는 저장된 내 캠퍼스를 학식 기본 캠퍼스로 사용할 수 있다

**우선순위:** P1

**Acceptance Criteria**

- 프로필에 캠퍼스가 저장되어 있으면 학식 화면의 기본 캠퍼스로 선택된다.
- 저장된 캠퍼스가 없으면 기존처럼 첫 번째 캠퍼스를 기본 선택한다.
- 사용자가 화면에서 캠퍼스를 바꾸면 현재 화면 조회 조건만 변경된다.

**Tasks**

- 프로필 조회 상태와 학식 화면 연동 방식 결정
- 학식 화면 초기 캠퍼스 선택 로직 수정
- 프로필 조회 실패 시 기존 기본값 fallback 처리

## Epic 5. 학사일정

### Story 5-1. 사용자는 앱에서 이번 달 학사일정을 조회할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 학사일정 화면 진입 시 이번 달 일정이 기본으로 표시된다.
- 일정은 날짜 또는 기간과 제목을 함께 보여준다.
- 해당 월에 일정이 없으면 일정 없음 상태를 보여준다.
- 학사일정을 불러오지 못하면 오류 상태를 보여준다.

**Tasks**

- 앱용 학사일정 API 설계: `GET /schedules?month=`
- response DTO에 연도, 월, 일정 목록 포함
- 백엔드 native controller 추가
- `mobile/services/scheduleApi.ts` 추가
- `mobile/app/(tabs)/schedule.tsx` 화면 추가
- 이번 달 기본 조회 구현
- 일정 없음/호출 실패 상태 구현
- 학사일정 조회 테스트 추가

### Story 5-2. 사용자는 조회할 월을 선택해 학사일정을 볼 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 사용자는 1월부터 12월까지 월을 선택할 수 있다.
- 월을 선택하면 해당 월의 학사일정으로 화면이 갱신된다.
- 올바르지 않은 월은 앱 UI에서 선택할 수 없다.

**Tasks**

- 월 선택 segmented control, picker, 또는 horizontal selector 구현
- 월 변경 시 API 재조회
- 선택 월 유지 상태 구현
- 월 선택 UI 테스트 추가

### Story 5-3. 사용자는 오늘 이후 가까운 학사일정을 빠르게 확인할 수 있다

**우선순위:** P2

**Acceptance Criteria**

- 이번 달 일정 중 오늘 이후 가장 가까운 일정을 강조한다.
- 이미 지난 일정과 앞으로 남은 일정을 시각적으로 구분한다.

**Tasks**

- 일정 날짜 파싱 정책 정의
- 지난 일정/다가오는 일정 구분 로직 추가
- 가까운 일정 강조 UI 추가

## Epic 6. 셔틀버스

### Story 6-1. 사용자는 앱에서 셔틀 노선 목록을 조회할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 앱은 현재 제공 중인 `가좌 → 칠암 캠퍼스`, `칠암 → 가좌 캠퍼스` 노선을 보여준다.
- 첫 번째 노선이 기본 선택된다.
- 노선 정보를 불러오지 못하면 오류 상태를 보여준다.

**Tasks**

- 기존 셔틀 화면의 시나리오 충족 여부 점검
- 노선 없음 상태 추가
- 노선 API 실패 상태 재시도 추가

### Story 6-2. 사용자는 선택한 노선의 시간표와 다음 버스를 볼 수 있다

**우선순위:** P0

**Acceptance Criteria**

- 선택한 노선의 시간표가 구간별로 표시된다.
- 현재 시간 이후 가장 가까운 다음 버스 1개가 강조된다.
- 이미 지난 시간, 다음 버스 시간, 앞으로 남은 시간이 구분된다.
- 오늘 남은 운행이 없으면 “오늘 운행이 종료되었습니다”를 보여준다.
- 금요일에 운행하지 않는 시간은 별도 메모로 구분된다.

**Tasks**

- 기존 셔틀 화면의 다음 버스/시간표 표시 로직 점검
- 금요일 미운행 표시의 UI 명확성 개선
- 시간표 없음 상태 추가
- 셔틀 시간표 화면 테스트 추가

### Story 6-3. 앱은 셔틀 시간표를 최신 상태로 유지한다

**우선순위:** P0

**Acceptance Criteria**

- 시간표는 1분마다 자동으로 다시 조회된다.
- 사용자는 갱신 버튼을 눌러 즉시 다시 조회할 수 있다.
- 마지막 갱신 시점을 볼 수 있다.

**Tasks**

- 기존 polling 동작 점검
- 화면 이탈 시 polling 정리 여부 테스트
- 수동 갱신 중 중복 요청 방지

### Story 7-1. 앱은 native API 호출에 필요한 인증을 처리할 수 있다

**우선순위:** P0

**Acceptance Criteria**

- API request helper에 인증 헤더 추가
- 401 응답 처리 정책 구현
- 개발/운영 환경의 `EXPO_PUBLIC_API_URL` 설정 점검

- native controller는 앱이 사용하기 쉬운 JSON DTO를 반환한다.
- 공통 비즈니스 로직은 service에서 재사용한다.

**Tasks**

- 공지사항 native controller 추가
- 학사일정 native controller 추가
- 사용자/단과대학/학과 native controller 추가
- DTO와 Swagger 문서 정리
- 기존 챗봇 API 회귀 테스트 실행

## Epic 8. 품질과 안정성

### Story 8-1. 앱 주요 기능은 회귀 테스트로 보호된다

**우선순위:** P1

**Acceptance Criteria**

- 셔틀 다음 버스 계산 기준이 테스트된다.
- 공지사항/학사일정/프로필 API DTO가 테스트된다.

- 백엔드 native controller 테스트 추가

- 재진입 시 필요한 데이터가 다시 조회된다.

**Tasks**

- 화면별 refresh 정책 정리
- pull-to-refresh 적용 검토
- 캐시 또는 stale data 표시 정책 검토
- 오류 로깅/Sentry 연동 확인

## Epic 9. Android 출시 준비

### Story 9-1. 운영용 Android 앱 빌드 설정을 준비한다

**우선순위:** P0

**Acceptance Criteria**

- Android 패키지명이 확정되어 `mobile/app.json` 또는 Expo 설정에 반영된다.
- 앱 이름, 아이콘, 스플래시 이미지, 버전, 빌드 번호가 운영 기준으로 설정된다.
- 운영 API 주소가 Android release 빌드에 주입된다.
- Android App Bundle(`.aab`)을 생성할 수 있다.

**Tasks**

- Android 패키지명 확정
- `app.json`의 Android 설정 점검 및 수정
- 앱 이름, 앱 아이콘, adaptive icon, splash asset 최종 적용
- `EXPO_PUBLIC_API_URL` 운영 환경값 정리
- `eas.json` production profile 점검
- `eas build --platform android --profile production` 빌드 절차 문서화
- 빌드 산출물 다운로드 및 보관 위치 정리

### Story 9-2. Android 앱 서명과 배포 권한을 준비한다

**우선순위:** P0

**Acceptance Criteria**

- Google Play Console에 앱을 생성할 수 있는 개발자 계정이 준비된다.
- Android 앱 서명 키 관리 방식이 확정된다.
- Play App Signing 사용 여부가 정해진다.
- 릴리스 빌드가 동일한 패키지명과 서명 체계로 재빌드 가능하다.

**Tasks**

- Google Play Console 개발자 계정 상태 확인
- Play Console 앱 생성
- Expo/EAS credentials 설정 확인
- Android keystore 관리 방식 결정
- 서명 키 백업 및 접근 권한 관리 방식 문서화
- 앱 번들 업로드 가능 여부 사전 확인

### Story 9-3. Play Store 등록 정보를 준비한다

**우선순위:** P0

**Acceptance Criteria**

- Play Store에 필요한 앱 기본 정보가 작성된다.
- 앱 설명, 짧은 설명, 카테고리, 연락처 정보가 준비된다.
- Android 스크린샷과 앱 아이콘이 Play Store 규격에 맞게 준비된다.
- 개인정보처리방침 URL이 등록 가능하다.

**Tasks**

- 앱 이름과 짧은 설명 작성
- 상세 설명 작성
- 앱 카테고리와 태그 선정
- 개발자 연락처 이메일 입력값 확정
- 휴대전화 스크린샷 촬영
- 7인치/10인치 태블릿 스크린샷 필요 여부 확인
- 512x512 고해상도 아이콘 준비
- feature graphic 필요 여부 확인 및 제작
- 개인정보처리방침 페이지 준비 및 URL 등록

### Story 9-4. Android 권한, 데이터 보안, 정책 항목을 점검한다

**우선순위:** P0

**Acceptance Criteria**

- 앱에서 사용하는 Android 권한이 실제 기능에 필요한 최소 범위로 제한된다.
- Play Console 데이터 보안 섹션에 입력할 수집/공유 데이터 항목이 정리된다.
- 개인정보처리방침 내용과 앱 동작이 일치한다.
- 학교 공식 앱으로 오해되지 않도록 표현이 점검된다.

**Tasks**

- `app.json` 및 native permission 설정 확인
- 네트워크 통신, 사용자 식별자, 프로필 정보 사용 여부 정리
- 데이터 수집/공유/암호화/삭제 가능 여부 표 작성
- Play Console Data safety 문항 답변 초안 작성
- 개인정보처리방침에 캠퍼스/학과 설정 데이터 처리 내용 반영
- 앱 설명과 화면 문구에서 학교 공식 서비스 오인 가능성 점검

### Story 9-5. Android 내부 테스트 릴리스를 배포한다

**우선순위:** P0

**Acceptance Criteria**

- Play Console 내부 테스트 트랙에 `.aab`가 업로드된다.

- 발견된 P0/P1 이슈가 정리되고 수정 여부가 추적된다.

**Tasks**

- 내부 테스트 트랙 생성
- 테스터 이메일 목록 등록
- 첫 Android App Bundle 업로드
- 릴리스 노트 작성
- 내부 테스트 링크 공유
- Android 실제 기기 smoke test 실행
- 학식, 셔틀, 공지, 학사일정, 내 정보 기능 점검
- 네트워크 오류/빈 상태/인증 실패 케이스 점검
- 테스트 결과와 수정 이슈 기록

### Story 9-6. Android 프로덕션 출시를 진행한다

**Tasks**

- 출시 후보 버전 확정
- 프로덕션 릴리스 노트 작성
- Play Console 필수 정책/콘텐츠 등급/타겟 연령 항목 완료
- 프로덕션 트랙에 릴리스 제출
- 검토 반려 시 대응 항목 기록 및 수정
- 승인 후 배포 상태 확인
- 설치 후 첫 실행 smoke test 진행
- 출시 버전 태그 또는 릴리스 기록 작성

### Story 9-7. Android 출시 후 운영 모니터링을 준비한다

**우선순위:** P1

**Acceptance Criteria**
