# GitHub PR 본문 작성

## 작업 흐름

1. 사용자 입력을 해석합니다.
   - 입력이 있으면 대상 브랜치명 또는 Epic 식별자로 사용합니다.
   - 입력이 없으면 현재 체크아웃 브랜치를 기준으로 합니다.
   - 대상 브랜치가 명시되지 않으면 `main`을 기본값으로 삼습니다.
2. 저장소의 PR 템플릿을 읽습니다.
3. docs 폴더에 있는 백로그 문서를 읽습니다.
4. Git 기준 차이를 확인합니다.
   - 커밋: `git log <target>..HEAD --oneline`
   - 변경: `git diff <target>...HEAD`
   - 대상 브랜치가 로컬에 없으면 `origin/<target>`도 확인합니다.
5. 실제 변경과 도메인 흐름을 요약합니다.
   - 처음 보는 사람도 이해할 수 있도록 도메인/사용자 흐름을 한 단계 추상화해 설명합니다.
   - 구현 파일, 클래스, 함수명을 길게 나열하지 않습니다.
   - 핵심 변경, 검증 관점, 영향 범위를 PR 리뷰어가 판단할 수 있게 씁니다.
6. 템플릿의 섹션 구조와 체크리스트를 유지해 본문을 채웁니다.
   - 체크리스트 문구와 순서는 보존합니다.
   - 관련 이슈는 실제 번호가 확인될 때만 `close #번호`를 사용합니다.
   - 번호가 없고 템플릿이 이슈 항목을 요구하면 `없음` 또는 `해당 없음`으로 씁니다.
7. 사용자가 draft PR을 요청하면 GitHub PR 본문을 작성합니다.
   - GitHub MCP PR 작성 도구가 있으면 우선 사용합니다.
   - 없으면 인증된 GitHub CLI를 사용합니다.

## 출력 규칙

- 모든 내용은 한국어로 작성합니다.
